# 后端优化与去 n8n 重构方案（Vercel 部署）

> 目标：把现有“Next.js API → n8n 编排/执行 → webhook 回写 → 前端轮询”的模式，升级为“Next.js API（统一入口）→ 直连各家 Provider API（工程化 Client/Adapter）→（Provider webhook 或定时 Poller）回写 DB → 前端轮询”的可维护架构。

## 1. 现状梳理（当前项目的真实形态）

### 1.1 典型链路（以 Veo3/Hailuo/Baby/LipSync/EarthZoom 为代表）
1) `generate` 路由：鉴权 + 校验 +（可选）R2 上传 + RPC 落库/扣费 → 调用 n8n webhook 异步执行  
2) `webhook/*-ready` 路由：n8n 回调（共享密钥校验）→ 写入表的 `status/video_url/error_message`  
3) 前端：定时轮询 `status/[jobId]`，直到 `completed/failed`

你仓库里多条 API 路由都遵循这个模式：
- `src/app/api/veo3/generate/route.ts`：RPC 后调用 `N8N_WEBHOOK_URL_VEO3`
- `src/app/api/hailuo/generate/route.ts`：RPC 后调用 `N8N_HAILUO_WEBHOOK_URL`
- `src/app/api/baby/generate/route.ts`：R2 上传 + DB 写入 + 扣费后调用 n8n（且带硬编码 fallback URL，风险较高）
- `src/app/api/lipsync/generate/route.ts`、`src/app/api/earth-zoom/generate/route.ts`：同类结构
- `src/app/api/webhook/*-ready/route.ts`：用于接收 n8n 回调并更新数据库

### 1.2 已经存在的“直连”雏形（Seedance）
`src/app/api/seedance/generate/route.ts` 已经改为直连 `VolcengineClient`（`src/lib/volcengine-client.ts`），但当前实现里存在一个关键问题：
- 在请求返回给用户后，启动了 `pollTaskInBackground(...)` 做后台轮询。
- **在 Vercel/Serverless 环境中，请求结束后进程可能随时被回收**，后台轮询不保证执行，容易造成任务永远停在 `processing`。

结论：要彻底去 n8n，并保持异步任务可靠，需要引入一个“可靠执行器”（Provider webhook + 兜底轮询 Poller；或仅轮询 Poller）。

---

## 2. 总体目标架构（不依赖 n8n 的工程化方案）

### 2.1 统一角色划分
- **API 路由（请求入口）**：只负责鉴权/校验/落库/扣费/提交任务（submit），不做长耗时等待。
- **Provider Client/Adapter（直连层）**：封装各家 API 差异（鉴权、请求结构、状态映射、输出解析）。
- **Job 执行器（完成态推进）**：
  - 优先：Provider 支持 webhook 则用 webhook 推进状态；
  - 兜底：定时 Poller 扫描 `processing` 任务，调用 `getStatus` 推进到 `completed/failed`。

### 2.2 Vercel 部署下的推荐执行器实现
推荐：**Vercel Cron Jobs + 内部 Poller API**
- 增加一个内部路由（例如 `src/app/api/internal/poller/route.ts`），用 `INTERNAL_POLLER_SECRET` 保护。
- 在 `vercel.json` 或 Vercel 控制台配置 Cron（例如每 1 分钟触发一次），由 Cron 调用 Poller API。
- Poller 的职责：按工具表逐批扫描 processing job → 对应 provider client 查询状态 → 更新数据库 → 记录最后查询时间与错误次数（便于退避/熔断）。

这样你不需要在“用户请求”里后台轮询，也不依赖 n8n 的异步能力。

## 3. 直连目标未确定时，应该怎么设计（回答“直连的目标分别是什么？”）

你现在的情况是：每个工具未来可能对接不同 Provider，且未最终选型。正确做法是：

### 3.1 先确定“契约”，再逐个实现
为每类能力定义统一接口（示例）：

- `submit(input) -> { externalTaskId: string }`
- `getStatus(externalTaskId) -> { status: 'processing'|'completed'|'failed'; outputUrl?: string; errorMessage?: string }`
- （可选）`cancel(externalTaskId)`

然后每个 Provider 做一个实现，比如：
- `providers/veo3/*`、`providers/hailuo/*`、`providers/volcengine/*`、`providers/replicate/*`……

### 3.2 Provider 选型 checklist（落地前必须明确的点）
对每个工具（Veo3/Hailuo/LipSync/EarthZoom/Baby）逐项确认：
- **异步/同步**：提交后是否立即返回结果？还是返回 task id？
- **状态查询接口**：是否提供 `GET /tasks/:id` 之类接口？频率限制是什么？
- **Webhook**：是否支持结果回调？签名如何验？失败重试策略是什么？
- **输出资产**：结果是 URL 还是二进制？需要你再转存到 R2 吗？
- **计费与用量**：能否拿到真实用量（时长/tokens/秒数）？决定“扣费时机”（见 6.2）
- **限流与并发**：单用户并发限制/全局 QPS 限制（决定是否要队列）

在选型未定的阶段，你只需要把“契约 + DB 字段 + Poller”先搭好；具体 Provider 接入可以逐个实现。

## 4. 数据库与状态机优化（去 n8n 后会更重要）

### 4.1 建议统一 Job 状态机
建议所有生成表（`*_generations`）收敛成一致状态（哪怕底层表不同，也要一致语义）：
- `queued`（可选）
- `processing`
- `completed`
- `failed`

### 4.2 建议统一字段（最小集合）
每个 generation 表建议至少包含：
- `job_id`（你已有）
- `external_task_id`（直连 Provider 后强烈建议统一存在；Seedance 已有类似字段）
- `provider` / `model` / `version`（用于回放与排障）
- `status`、`output_url`（或 `video_url` / `generated_baby_url`）、`error_message`
- `created_at`、`completed_at`
- （可选）`last_polled_at`、`poll_attempts`、`next_poll_at`（退避策略）

如果你走 Effect 表驱动（建议），那么表里也应包含 `provider/model/api_endpoint/input_schema/validation_rules/processing_config` 这类字段，用于统一配置。

## 5. 代码结构建议（把重复逻辑工程化）

### 5.1 Provider 层（直连 API 的统一入口）
新增目录（示例）：
- `src/lib/providers/types.ts`：统一的输入/输出类型、状态映射
- `src/lib/providers/registry.ts`：`createProviderClient(providerName, config)`
- `src/lib/providers/*Client.ts`：各家 Provider 的 submit/getStatus 实现

### 5.2 抽一个通用的 “submit flow” 模板（减少 API 路由重复）
你现在每个 `generate` 路由都在重复：
- 鉴权（supabase）
- 参数校验（长度/枚举/必填）
- R2 上传（多套实现，桶/hostname/clean name 逻辑不一致）
- RPC 落库/扣费（多个函数，返回 status 各不相同）
- 异步提交（n8n 或直连）

建议分层：
- `src/lib/auth/*`：统一拿用户
- `src/lib/validation/*`：用 Zod/Schema 做请求校验（更稳定）
- `src/lib/storage/r2.ts`：唯一的 R2 上传实现（所有工具复用）
- `src/lib/db/*`：封装 RPC 调用与标准化返回（例如统一成 `ok | insufficient_credits | concurrent_generation_exists | ...`）
- `src/lib/providers/*`：直连提交与查询

### 5.3 Webhook 路由的收口
你已经有 `src/lib/supabaseAdmin.ts`，但 webhook 里又手写初始化。建议全部改为复用 `supabaseAdmin`，避免环境变量判断与日志重复。

## 6. 计费与一致性（去 n8n 之后必须明确）

### 6.1 当前计费模式概览
当前多数工具是在 `generate` 时 RPC 里“先扣费/或预扣费”，然后 n8n 回调只更新状态。
也存在“完成后按用量扣费”的设计痕迹：`src/app/api/webhook/n8n-video-ready/route.ts` 里调用了 `deduct_credits_by_duration`（说明曾经尝试按实际 duration 扣费）。

### 6.2 去 n8n 后的两种计费策略（需要你选一个）
A) **预扣费（最省事，和现状一致）**
- generate：RPC 创建记录并扣费
- poller/webhook：只更新结果
- 失败退款：可选（失败时退积分需要额外逻辑）

B) **完成后按用量扣费（更公平，但工程复杂）**
- generate：只创建记录不扣费（或冻结额度）
- 完成：poller/webhook 获得 duration/tokens → 调 `deduct_credits_by_duration` 或类似函数
- 风险：如果 provider 不给用量、或回调丢失，需要兜底

建议：第一阶段先用 A 跑通直连迁移；第二阶段再把计费升级为 B（如果业务确实需要）。

## 7. 分阶段迁移计划（Vercel 友好 + 可回滚）

### 阶段 0：准备（不改业务行为）
- 新增 Provider 抽象接口与 registry（空实现或仅 Seedance）
- 新增 R2 上传统一模块，逐步替换各路由内的 `uploadToR2`
- 新增内部 Poller API（先不启用 Cron）
- 加统一错误响应结构（至少统一 `error_code` 字段，便于前端处理）

### 阶段 1：以 Seedance 为试点（你已经直连了一半）
- 把 `pollTaskInBackground` 从 `seedance/generate` 移除
- 改为：generate 只 submit 并写 `external_task_id`
- 由 Poller 推进 completed/failed
- 配置 Vercel Cron 调用 Poller

### 阶段 2：迁移 Veo3/Hailuo/EarthZoom/LipSync/Baby（逐个）
对每个工具做同一类改造：
- `generate`: “n8n webhook 调用”替换为 `provider.submit`
- 写入 `external_task_id`
- 删除/废弃对应的 `webhook/*-ready`（先灰度：保留但不再依赖）
- 确保 `status/[jobId]` 不变（前端无感）

### 阶段 3：去除 n8n 相关配置与回调
- 删除环境变量 `N8N_*_WEBHOOK_URL`（或不再依赖）
- 删除 n8n 回调路由（或改成 provider webhook 路由，签名校验换成 provider 的规范）
- 清理硬编码的 webhook fallback（Baby 的那个必须去掉）

## 8. 额外“后端工程化”优化清单（建议纳入后续迭代）

### 8.1 可靠性
- Poller 退避：`next_poll_at` + `poll_attempts`，避免高频打爆 provider
- 幂等：对 `completed` 更新做“只从 processing→completed”更新，避免重复回写（你部分 webhook 已做）
- 统一超时与重试：封装 `fetchWithTimeout` + 可配置重试（仅对可重试错误）

### 8.2 安全
- 内部 Poller API 必须用单独 secret（不要复用 `N8N_API_KEY`）
- Provider 密钥与回调密钥分离（职责单一、最小暴露面）
- 入参校验全面收敛到 schema（Zod），减少手写 if/else 漏洞面

### 8.3 可观测性
- 每个 job 生成 `trace_id`（或复用 `job_id`）贯穿日志
- Provider 请求日志：记录 providerName、external_task_id、耗时、返回状态（脱敏）

### 8.4 维护成本
- 抽统一 `GenerationService`：`createJob` / `submitJob` / `queryJob` / `updateJob`
- 尽量收敛表结构与字段命名（否则每加一个工具都要复制一套 status API）

---

## 9. 下一步你需要决定的最少事项（不影响先开工）

即使直连目标未定，也建议先做以下最少决策，便于开工：
- 是否接受“Poller + Cron”作为异步执行的主方案（Vercel 推荐是：**是**）
- 先采用哪种计费策略（推荐第一阶段：**预扣费**）
- 每个工具的 Provider 先留空还是用一个临时 Provider（例如都走同一家聚合 API）过渡
